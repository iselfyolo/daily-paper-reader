## 摘要
模拟学习者在教育系统中的行为和反应的人工模型，是用于大规模评估辅导策略和反馈机制的一种极具前景的工具。然而，编程教育中现有的许多方法依赖于提示大型专有语言模型，这引发了关于隐私、成本和依赖性的担忧。在这项工作中，我们提出了一种利用真实学生过程数据训练开源权重人工编程学习者的方法。我们的方法将时间日志轨迹序列化为对话格式，将每个学生的解题过程表示为学习者与其自动评估系统之间的对话。学生提交的代码和环境反馈（如测试结果、成绩和错误追踪）构成了交替的对话轮次，使模型能够从迭代调试过程中学习。此外，我们引入了一种结合监督微调与偏好优化的训练流水线，以使模型与真实的学生调试行为保持一致。我们通过在 Python 编程作业的大规模真实学生提交数据集上训练 4B 和 8B 规模的 Qwen 模型来评估我们的框架。结果表明，整合环境反馈增强了模型复制学生调试行为的能力，在功能对齐和代码相似度方面优于先前的仅代码方法以及基于提示的大型语言模型基准。我们发布了代码以支持可复现性。

## Abstract
Artificial models that simulate how learners act and respond within educational systems are a promising tool for evaluating tutoring strategies and feedback mechanisms at scale. However, many existing approaches in programming education rely on prompting large, proprietary language models, raising concerns around privacy, cost, and dependence. In this work, we propose a method for training open-weight artificial programming learners using authentic student process data. Our approach serializes temporal log traces into a conversational format, representing each student's problem-solving process as a dialogue between the learner and their automated assessment system. Student code submissions and environment feedback, such as test outcomes, grades, and error traces, form alternating conversational turns, enabling models to learn from the iterative debugging process. We additionally introduce a training pipeline combining supervised fine-tuning with preference optimization to align models with authentic student debugging behavior. We evaluate our framework by training Qwen models at 4B and 8B scales on a large-scale dataset of real student submissions to Python programming assignments. Our results show that incorporating environment feedback strengthens the models' ability to replicate student debugging behavior, improving over both prior code-only approaches and prompted large language models baselines in functional alignment and code similarity. We release our code to support reproducibility.

#### 1. 核心问题与研究动机
*   **核心问题**：如何在不依赖昂贵、封闭且存在隐私风险的专有大模型（如 GPT-4）的前提下，利用开源模型真实地模拟编程初学者的解题和调试行为。
*   **研究动机**：
    *   **教育需求**：大规模评估教学策略和反馈机制需要“人工学生”作为模拟对象，以避免频繁的课堂实验成本。
    *   **现有局限**：目前的模拟方法多基于 Prompting，往往只能复制表面的错误模式，难以捕捉学生在面对系统反馈时动态、迭代的调试过程。
    *   **数据利用**：编程学习平台积累了大量的“过程数据”（代码提交+系统反馈），但尚未被充分用于训练小规模的专用模拟模型。

#### 2. 方法论
*   **核心思想**：将学生的编程轨迹（Trajectory）转化为**对话式序列化（Conversational Serialization）**格式，将解题过程建模为“学生”与“自动评测系统（Autograder）”之间的多轮对话。
*   **关键技术细节**：
    *   **角色映射**：将“自动评测系统”设为 **User**（提供题目、测试结果、错误追踪），将“学生”设为 **Assistant**（提交代码）。
    *   **训练流水线**：
        1.  **监督微调 (SFT)**：在序列化的对话数据上进行微调，使模型学习学生在特定反馈下的代码演变模式。
        2.  **离线偏好优化 (DPO)**：构建偏好对。**正例**是学生实际的下一次提交；**负例**是学生后续轨迹中第一个得分不同的提交。这旨在防止模型“过早”生成正确答案，从而更真实地模拟学生的挣扎过程。
        3.  **在线偏好优化 (DAPO)**：基于 GRPO 的变体，通过采样多个候选代码，并根据功能对齐（如 AST 匹配、得分一致性）给予奖励。
*   **算法流程**：输入历史提交和反馈 -> 模型生成下一阶段代码 -> 环境反馈结果 -> 循环迭代（Rollout）。

#### 3. 实验设计
*   **数据集**：使用 **FalconCode**，这是一个包含多年 Python 入门课程提交记录的大规模数据集。实验选取了其中中等难度的“Lab”作业。
*   **Benchmark 与对比方法**：
    *   **BASE**：未经微调的 Qwen 原始模型。
    *   **PARA**：仅使用代码历史进行训练（不含环境反馈），代表先前的研究路径。
    *   **SFT/DPO/DAPO**：本研究提出的不同训练阶段的模型。
    *   **GPT-5-mini**：作为高性能闭源模型的基准。
*   **评估指标**：
    *   **Coverage（覆盖率）**：模型是否能持续生成代码直到学生完成任务（防止过早提交正确答案）。
    *   **Grade Proximity（成绩接近度）**：模型生成的代码得分与真实学生得分的接近程度。
    *   **CodeBLEU**：衡量代码在语法和语义上的相似度。

#### 4. 资源与算力
*   **硬件设施**：使用 **NVIDIA V100 GPU (32 GB VRAM)** 集群。
*   **训练框架**：采用 **Unsloth** 库进行加速，并使用 **QLoRA**（秩 r=8, α=16）进行高效微调。
*   **训练细节**：上下文窗口设置为 4096 token，采用动态滑动窗口处理超长轨迹。

#### 5. 实验数量与充分性
*   **实验规模**：针对 **Qwen3-4B** 和 **Qwen3-8B** 两种规模的模型进行了完整实验。
*   **数据划分**：使用 2021 春季数据训练，2022 春季数据（1000 条独立轨迹）测试，保证了时间维度上的泛化性验证。
*   **充分性**：实验涵盖了从单步预测到 5 步自动回归（Rollout）的性能评估，并对比了有无环境反馈、不同优化算法的效果，实验设计较为客观、公平。

#### 6. 主要结论与发现
*   **环境反馈是关键**：整合了 Autograder 反馈的模型（SFT）在所有指标上均优于仅看代码历史的模型（PARA），证明反馈数据对模拟调试行为至关重要。
*   **DPO 效果显著**：DPO 训练的模型在“覆盖率”和“成绩接近度”上表现最好，能够最真实地追踪学生的进度，有效缓解了模型直接跳向正确答案的倾向。
*   **闭源模型不适用**：GPT-5-mini 虽然代码能力强，但无法模拟初学者的渐进式错误，其生成的代码与学生真实轨迹的相似度最低。
*   **模型规模影响**：8B 模型在代码相似度上略优于 4B，但在模拟学生行为逻辑上，4B 结合 DPO 已能达到极高水平。

#### 7. 优点
*   **格式创新**：将复杂的编程日志巧妙地转化为标准对话格式，降低了利用现有 LLM 架构的门槛。
*   **对齐策略有效**：通过 DPO 成功解决了模拟器“过于聪明”的问题，使其能够模拟“挣扎”的过程。
*   **实用性强**：使用中等规模的开源模型，便于教育机构在保护隐私的前提下本地部署。

#### 8. 不足与局限
*   **数据集局限**：仅在 Python 语言和单一机构的数据集上验证，跨语言和跨文化的泛化性有待测试。
*   **样本偏差**：测试集中 95% 的学生最终都成功完成了作业，模型对于“彻底失败”或“中途放弃”的学生行为模拟不足。
*   **缺乏推理链**：模型直接生成代码，缺乏类似思维链（CoT）的中间推理过程，这可能限制了其对复杂逻辑错误的解释力。
*   **下游验证缺失**：尚未实际测试这些“人工学生”在训练真实辅导机器人（Tutor AI）时的实际增益。
