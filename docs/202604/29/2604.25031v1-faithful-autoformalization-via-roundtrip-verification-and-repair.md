## 摘要
当大语言模型（LLM）将自然语言形式化时，我们如何确保输出是忠实的？我们提出了一种无需基准真相标注的往返验证方法：首先将陈述形式化，接着将结果翻译回自然语言并重新形式化，最后使用形式化工具检查逻辑等价性。当两次形式化结果一致时，即证明了形式化的忠实性。若不一致，诊断步骤将识别失败的翻译阶段，并由针对性的修复算子尝试纠正。我们在 150 条交通规则上使用 Claude Opus 4.6 和 GPT-5.2 对该方法进行了评估。结果显示，诊断引导的修复将两个模型的形式等价性从 45-61% 提升至 83-85%，优于随机修复基准。一项独立的自然语言推理（NLI）分析证实，形式等价性与较少的语义漂移具有相关性。

## Abstract
When an LLM formalizes natural language, how do we know the output is faithful? We propose a roundtrip verification approach which does not require ground-truth annotations: formalize a statement, translate the result back to natural language, re-formalize, and use a formal tool to check logical equivalence. When the two formalizations agree, this provides evidence of a faithful formalization. When they disagree, a diagnosis step identifies which translation stage failed, and a targeted repair operator attempts to correct that stage. We evaluate our approach on 150 traffic rules using Claude Opus 4.6 and GPT-5.2. Diagnosis-guided repair raises formal equivalence from 45--61% to 83--85% for both models, outperforming a random-repair baseline. An independent NLI analysis confirms that formal equivalence is correlated with less semantic drift.

### 论文总结：通过往返验证与修复实现忠实的自动形式化

#### 1. 核心问题与整体含义（研究动机和背景）
自动形式化（Autoformalization）是将自然语言（NL）转换为机器可检查的形式化表示（如逻辑公式、代码或证明脚本）的过程。目前该领域面临的核心挑战是：**在没有人工标注的“基准真相”（Ground-truth）的情况下，如何验证 LLM 生成的形式化结果是否忠实地表达了原始文本的语义？** 现有的评估方法通常依赖于语法正确性或下游任务的成功率，但这无法直接测试语义的忠实性。此外，当出现错误时，现有的修复手段往往是盲目的全局重新生成，缺乏对错误来源的精准诊断。

#### 2. 论文提出的方法论
论文提出了一种名为**往返验证与修复（Roundtrip Verification and Repair）**的框架，其核心思想是通过闭环翻译来产生验证信号：
*   **往返流水线（Three Stages）**：
    1.  **$T_1$（形式化）**：将原始自然语言 $x$ 翻译为形式化表达式 $y_{orig}$。
    2.  **$T_2$（回译）**：将 $y_{orig}$ 翻译回自然语言 $x'$。
    3.  **$T_3$（再形式化）**：将 $x'$ 再次翻译为形式化表达式 $y_{rt}$。
*   **等价性检查**：利用形式化工具（如 SMT 求解器 Z3）检查 $y_{orig}$ 与 $y_{rt}$ 是否在逻辑上等价（$y_{orig} \equiv_\Sigma y_{rt}$）。如果等价，则为忠实性提供了证据；如果不等价，则触发修复流程。
*   **诊断与定向修复**：
    *   **诊断（Diagnosis）**：由 LLM 充当裁判，按顺序检查 $T_1, T_2, T_3$ 阶段，识别第一个引入语义偏差的阶段，并生成解释。
    *   **定向修复（Scoped Repair）**：根据诊断结果，仅针对故障阶段进行修复（注入诊断反馈），并重新生成该阶段之后的所有产物。
*   **辅助语义信号**：引入自然语言推理（NLI）模型，通过双向蕴含检测原始文本 $x$ 与回译文本 $x'$ 之间的语义漂移，作为独立于形式化工具的验证补充。

#### 3. 实验设计
*   **数据集/场景**：使用了从《德克萨斯州交通法典》（Texas Transportation Code）中提取的 **150 条交通规则**。该领域具有法律条文的严谨性，且适合转化为一阶逻辑。
*   **目标形式化语言**：多排序一阶逻辑（Many-sorted First-order Logic），使用 SMT-LIB 标准。
*   **对比模型**：使用了两个顶尖模型 **Claude Opus 4.6** 和 **GPT-5.2**。
*   **基准方法（Baselines）**：
    1.  **No Repair**：仅运行一次往返流程，不进行修复。
    2.  **Random Repair**：在不一致时随机选择一个阶段进行修复，不使用诊断信息。
    3.  **Full (Ours)**：使用诊断引导的定向修复。

#### 4. 资源与算力
*   **LLM 算力**：实验完全基于商业 API（Anthropic 和 OpenAI），总计进行了超过 2000 次 LLM 调用。
*   **本地算力**：SMT 求解（Z3）和 NLI 推理（BART-large-MNLI）在普通笔记本电脑 CPU（Apple M 系列）上运行，所有规则的验证在 10 分钟内完成。
*   **训练时长**：该研究不涉及模型训练或微调，主要成本在于推理延迟和 API 调用限制。

#### 5. 实验数量与充分性
*   **实验规模**：针对 150 条规则，在 2 个模型和 3 种修复策略下进行了多轮迭代（最多 3 次修复尝试）。
*   **充分性**：实验设计了消融实验（对比随机修复），证明了诊断信息的有效性。同时，通过 NLI 进行了跨模态的验证，增强了结论的客观性。
*   **公平性**：所有模型使用相同的 Prompt 模板和领域 Schema，实验条件受控。

#### 6. 主要结论与发现
*   **修复效果显著**：诊断引导的修复将 Claude 的形式等价率从 44.7% 提升至 **85.3%**，将 GPT 的从 61.3% 提升至 **82.7%**。
*   **诊断效率高**：相比随机修复，诊断引导的方法在 Claude 上以更低的平均成本（减少 41% 的单次修复调用成本）实现了近两倍的修复成功数。
*   **形式与语义相关性**：通过 NLI 分析发现，通过形式等价性检查的规则，其语义漂移显著低于未通过的规则，证明了往返验证作为忠实性指标的有效性。
*   **模型差异**：不同模型表现出不同的错误模式，GPT 倾向于在第一阶段（形式化）出错，而 Claude 的错误分布较均匀。

#### 7. 优点
*   **无需标注**：解决了自动形式化领域长期存在的“缺乏验证基准”的问题。
*   **形式化无关性**：该框架可推广至代码生成、证明助手（如 Lean）等任何具备等价性检查工具的领域。
*   **精准修复**：通过阶段性诊断避免了盲目重试，提高了修复效率和成功率。

#### 8. 不足与局限
*   **Schema 瓶颈**：手动设计的领域 Schema（本体论）限制了表达力。如果 Schema 缺失某些谓词，模型无论如何修复都无法达成忠实翻译。
*   **必要非充分条件**：往返一致性是忠实性的必要条件而非充分条件。理论上存在“系统性错误”导致流水线在错误的语义点达成一致（虽然 NLI 检查降低了此类风险）。
*   **领域局限**：目前仅在交通规则（法律条文）上验证，尚未测试其在复杂数学证明或大规模软件工程中的表现。
*   **依赖诊断质量**：修复效果高度依赖于 LLM 裁判的诊断准确性，若诊断错误，修复也会失效。
