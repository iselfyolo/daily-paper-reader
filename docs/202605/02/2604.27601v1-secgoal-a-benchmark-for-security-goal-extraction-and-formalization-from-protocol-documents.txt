Abstract 

Formal verification provides rigorous guaran-tees for cryptographic security, yet automat-ing the extraction and formalization of se-curity goals from natural language protocol documents remains a major bottleneck, com-pounded by the scarcity of expert-annotated resources and integrated frameworks bridg-ing unstructured text and symbolic logic. We introduce SecGoal, the first expert-annotated benchmark covering 15 widely deployed proto-col documents (e.g., 5G-AKA, TLS 1.3), and AIFG, an AI-assisted framework that decom-poses the task into context-aware goal extrac-tion and retrieval-augmented formalization. We conduct a comprehensive evaluation to assess whether contemporary LLMs are ready to au-tomate this pipeline. Our results reveal a pro-nounced precision-recall imbalance: frontier models (e.g., Gemini 2.5-Pro) achieve high re-call but precision below 15%, frequently mis-classifying operational text as security goals. In contrast, instruction tuning on SecGoal enables compact models (7B/9B) to achieve F1-scores above 80%, substantially outperforming larger general-purpose models. Our work establishes a foundational dataset and reproducible base-line for automated formal protocol analysis. 

1 Introduction 

Formal verification is critical for cryptographic se-curity, providing rigorous guarantees for widely de-ployed protocols such as TLS 1.3 (Bhargavan et al., 2017; Cremers et al., 2017) and 5G-AKA (Basin et al., 2018; Cremers and Dehnel-Wild, 2019). As illustrated in Figure 1, formal verification typically requires domain experts to manually curate docu-ments and encode protocol logic, threat models, and security goals—such as confidentiality and authentication—into code for tools like Tamarin (Meier et al., 2013) and ProVerif (Blanchet et al., 2018). Upon execution, these tools can automat-ically generate attack traces for subtle flaws that

evade conventional testing, or conversely, provide mathematical proofs that the protocol satisfies its intended security properties. Crucially, the effec-tiveness of formal verification depends on an in-termediate step that remains largely manual: ex-tracting security-relevant goals from protocol doc-uments and mapping them into precise symbolic descriptions of security properties. This step is challenging because protocol doc-uments, such as IETF RFCs, are written in long-form, unstructured natural language, while verifi-cation tools operate on precise security properties. However, security goals are rarely stated explic-itly; instead, they are dispersed across descriptive text, threat discussions, and design rationales. Even when identified, translating these goals into formal, tool-checkable property statements remains largely unsystematic, creating a persistent specification-to-model gap that limits the scalability of formal protocol analysis. Recent advances in large language models (LLMs) have sparked interest in autoformaliza-tion—the automated translation of natural language into formal specifications (Wu et al., 2022) and for-mal specification inference (Le-Cong et al., 2025). LLMs have demonstrated strong capabilities in code generation (Chen, 2021), mathematical proof synthesis (Jiang et al., 2022), and assertion genera-tion (Cosler et al., 2023), suggesting their potential to assist in automating security goal extraction. However, security protocols pose a particularly difficult setting: documents often span hundreds of pages, embed goals implicitly, and require long-context semantic reasoning, where errors can lead 1

to incomplete or unsound formal analyses. These observations raise a fundamental question: are LLMs ready to formalize security protocols from protocol documents? Answering this question is hindered by a lack of appropriate resources. Exist-ing work on LLMs for formal methods primarily focuses on code-level annotations or state-machine inference (Wael et al., 2025), while publicly avail-able datasets aligning natural language protocol documents with formal security properties are ab-sent. This data gap prevents systematic evaluation of LLM capabilities in this high-stakes setting and the development of LLM-based methods for auto-matic formal analysis of security protocols. To address this gap, we introduce SecGoal, the first expert-annotated dataset and benchmark specifically designed for security goal extraction from protocol documents. Building on SecGoal, we propose AIFG, a unified framework that inte-grates security goal extraction with the generation of precise formal security properties. Using this framework, we conduct the first comprehensive empirical study of automated security property ex-traction. Our results reveal a clear gap between promise and practice: LLMs, including state-of-the-art models such as GPT-4o and Gemini2.5-Pro, often achieve high recall but lack the precision required for formal analysis. We show that instruc-tion tuning on SecGoal substantially improves ex-traction quality and that, within AIFG, these capa-bilities can be further leveraged to generate precise formal security properties—representing a concrete step toward automated formal protocol analysis. Our main contributions are as follows: • We introduce SecGoal, the first expert-annotated dataset mapping natural language protocol docu-ments to structured security goal classifications and formalizable statements. This dataset fills a critical void at the intersection of NLP and formal verification. • We introduce AIFG, the first framework to unify security goal extraction with formal and precise security-property generation from natural lan-guage protocol documents. • We conduct the first systematic evaluation of LLMs for automated security goal extraction and formalization from protocol documents. 

2 Related Works 

LLMs for Autoformalization. A growing body of research leverages LLMs to translate natural lan-guage constraints into formal artifacts for general software and hardware verification. Tools such as NL2Spec (Cosler et al., 2023) and recent ap-proaches by Beg et al. (Beg et al., 2025) focus on generating code-level assertions (e.g., Dafny, JML) or function contracts (Pearce et al., 2023). Simi-larly, in hardware, AssertLLM (Yan et al., 2025) automates the generation of SystemVerilog Asser-tions. However, as highlighted by FormalBench (Le-Cong et al., 2025), general-purpose models struggle with the logical rigor required for veri-fication. Crucially, these approaches operate in deterministic contexts (e.g., source code or local-ized comments). They do not address the linguis-tic challenge of extracting global security goals from unstructured, ambiguous protocol documents, where the “code” does not yet exist. 

Automated Protocol Modeling. Prior work on protocol analysis typically assumes structured in-puts or focuses on operational flows rather than se-curity logic. One stream translates semi-formal no-tations (e.g., Alice & Bob) into verification models using compilers (Keller and Basin, 2014) or LLMs (Li et al., 2025). Regarding operational extraction, RFCNLP (Pacheco et al., 2022) and PROSPER (Sharma and Yegneswaran, 2023) utilize BERT-based architectures to extract Finite State Machines (FSMs). More recently, AutoSM (Mao et al., 2025) leveraged LLMs and intermediate lambda calculus to synthesize executable symbolic models. How-ever, these methods face a dual limitation: they either require manual intermediate translation or overlook the extraction of precise security proper-ties. 

Datasets and Benchmarks. Existing resources largely bypass the extraction challenge. Crypto-FormalEval (Curaba et al., 2024) benchmarks the translation of pre-defined properties into Tamarin code, assuming the extraction is already done. In the smart contract domain, PropertyGPT (Liu et al., 2024b) relies on source code structure and com-piler feedback, which are unavailable in natural language documents. To the best of our knowledge, SecGoal is the first dataset specifically designed for the extraction of security properties directly from raw protocol documents, filling a critical void at the intersection of NLP and formal verification. Complementing this resource, we introduce AIFG to establish a rigorous baseline for this novel task. 2Figure 2: Overview of the AIFG framework. 

3 The SecGoal Dataset 

This section presents SecGoal, detailing our pro-tocol selection criteria, data processing pipeline, human-in-the-loop annotation methodology, and final dataset statistics. 

3.1 Data Selection and Scope 

The SecGoal dataset encompasses a diverse array of 15 widely deployed protocols spanning criti-cal domains, including telecommunications (e.g., 5G-AKA), industrial IoT (e.g., OPC-UA, SPDM), and web security (e.g., TLS 1.3, FIDO2). Cru-cially, these documents are curated from heteroge-neous sources, extending beyond structured IETF RFCs to include broader industrial specifications and academic literature. Distinguishing itself from other synthetic datasets, SecGoal reflects the inher-ent complexity of real-world security engineering, where security goals are often implicitly embedded within verbose narrative descriptions. 

3.2 Data Preprocessing 

To mitigate context window limitations (Liu et al., 2024a) inherent in processing lengthy documents, we implemented a two-stage pipeline: 

Noise Reduction. We filtered out non-normative sections (e.g., bibliographies, tables of contents) to enhance the signal-to-noise ratio of documents. 

Context-Aware Segmentation. The text was di-vided into overlapping chunks. Crucially, we ap-plied a dynamic boundary extension strategy: in-stead of fixed-token truncation, chunk boundaries were extended to the nearest sentence delimiter. This prevents semantic fragmentation, ensuring se-curity goals remain self-contained. 

3.3 Human-in-the-Loop Annotation 

To ensure quality, we employed a Human-in-the-Loop (HITL) pipeline with three phases: • High-Recall Candidate Generation: We uti-lized an LLM with multiple queries to generate broad candidate security goals. The primary goal was to maximize coverage, surfacing potential goals from extensive narratives. This served as a preliminary reference, shifting human effort from raw extraction to precise verification. • Expert Refinement and Grounding: Three domain experts performed the core annotation with a dual objective: filtering hallucinations and recovering omissions. Crucially, experts did not rely solely on model candidates but grounded their validation in established formal analysis literature (e.g., TLS 1.3 (Bhargavan et al., 2017; Cremers et al., 2017), 5G-AKA (Basin et al., 2018; Cremers and Dehnel-Wild, 2019), and FIDO2 (Guan et al., 2022)). This cross-referencing helped identify subtle goals overlooked by the LLM and mitigate model bias. • Adjudication and Agreement Check: Two Ph.D. annotators in formal protocol analysis in-dependently reviewed all 15 protocols, using LLM candidates as a starting reference and sup-plementing missing goals from formal analysis literature, without access to each other’s labels. Disagreements were resolved by a senior arbitra-tor. Inter-annotator agreement yields a Jaccard of 

65.83% and Pairwise F1 of 79.39% . Full details are in Appendix C. 

3.4 Dataset Statistics and Split 

In total, the SecGoal dataset annotates over 100 distinct formal security properties corresponding to approximately 300–400 natural language security goal statements. A distinctive characteristic of Sec-Goal is its sparsity: valid security-goal chunks are buried within operational details, constituting less than 9% of the total text. This pronounced class im-balance challenges models to distinguish security goals from security implementation guidelines. A detailed breakdown of the distribution of secu-3rity versus non-security chunks for each protocol is provided in Appendix B. To rigorously evalu-ate generalization capabilities, we enforce a strict document-level split, holding out entire protocols (e.g., SPDM) for testing. 

4 AIFG Framework 

In this section, we present AIFG, a coarse-to-fine framework designed to extract and generate seman-tically faithful, formal security properties from un-structured protocol documents. 

4.1 Overview 

AIFG is a coarse-to-fine framework designed to bridge unstructured protocol documents and for-mal security properties. We formalize this task as mapping the raw protocol documents D to a set of canonical formal property descriptions 

P = {p1, . . . , p m}.Rather than attempting end-to-end translation, AIFG decomposes the pipeline into two stages: extraction and formalization. Guided by two key principles, our design separates semantic filtering from symbolic reasoning to explicitly handle the sparsity of security goals, and grounds generation in retrieved context to ensure symbolic consistency. As illustrated in Figure 2, Stage I ( Mext ) fil-ters operational noise from the protocol documents 

D to identify security goal statements G ⊂ C.Subsequently, Stage II ( Mf orm ) utilizes Retrieval-Augmented Generation (RAG) to translate these extracted goals into structured properties P. The following sections describe each stage in detail. 

4.2 Stage I: Security Goal Extraction 

Deriving formal security-property descriptions from complete protocol documents is computation-ally challenging and susceptible to context dilution. Therefore, this stage acts as a semantic filter: it scans text segments to detect the presence of secu-rity goals and, upon positive identification, extracts the precise security goal statements. 

Model Fine-Tuning. To achieve optimal perfor-mance, we instantiate the AIFG Extractor with a fine-tuned model, although the framework is de-signed to be model-agnostic. A critical challenge stems from the severe class imbalance in techni-cal specifications, where valid security goals are extremely sparse relative to background text. This distributional bias often causes the model to overfit to the majority class (non-goal content), hindering its ability to extract security goal sentences. To address this, we implement a Randomized Negative Downsampling strategy (Liu et al., 2008). By constructing batches with a balanced positive-to-negative ratio (e.g., 1:3), we force the model to learn a calibrated decision boundary, effectively distinguishing true security goal sentences from security-measure sentences and operational noise. 

Document Preprocessing. To ensure training-inference consistency, we employ the same prepro-cessing pipeline used in our data collection. Specif-ically, the raw protocol document D is segmented via a sliding window with overlap to preserve se-mantic continuity. This process yields a sequence of text chunks C = {c1, . . . , c n}, which are subse-quently fed into the Extractor. 

Goal Extraction. During inference, the instanti-ated AIFG Extractor processes each preprocessed chunk ci from the document D in a sequential man-ner. This process operates as an extraction mecha-nism governed by conditional generation logic. Initially, the model evaluates the semantic con-tent of ci to determine the presence of security goal statements. If the chunk contains no security goal statement, the model explicitly generates a prede-fined special token (e.g., [] ) to signify absence. Conversely, for chunks identified as containing se-curity goals, the model generates a structured list of all distinct security goal statements. Finally, the local extractions from all chunks are aggregated and de-duplicated to constitute the document-level candidate set G.

4.3 Stage II: Security Goal Formalization 

Translating the extracted security goal sentences 

G into accurate formal security properties is non-trivial. Unlike standard code generation, this task is plagued by specific linguistic and logical barriers: • Ambiguity and Context: Narrative goals fre-quently lack explicit entity references (e.g., omit-ting which specific nonce is used), necessitating deep contextual inference. • Symbolic Inconsistency: Accurate properties must strictly adhere to the variable namespace of the formal model. Generic terms (e.g., “server’s random number”) often fail to match specific protocol flow variables. • Syntactic and Mapping Complexity: There is a significant disconnect between verbose narra-tive descriptions and the abbreviated syntax of 4formal tools. Furthermore, the relationship is non-bijective, where a single narrative goal im-plies multiple distinct security properties. 

Contextual Retrieval. To systematically resolve ambiguity and context, we employ a RAG mecha-nism specifically for contextual grounding. Since entity definitions are often scattered throughout the full protocol documents D, naive translation leads to variable mismatches. We utilize each extracted goal g ∈ G as a semantic query to retrieve the top-k

relevant chunks ( Cret ) via a dense retriever. These chunks contain the precise definitions of data enti-ties implied in g, providing the necessary ground-ing for variable instantiation. 

Formal Synthesis. Subsequently, to address sym-bolic inconsistency and syntactic gaps, we inte-grate the protocol flow ( F)—a structured defini-tion of message exchanges—and a formal tem-plate schema ( S) alongside Cret into the generation process, formulated as p ∼ P (· | g, Cret , F, S).This mechanism enforces symbolic grounding by mapping ambiguous natural language terms to pre-cise, canonical variable symbols. For example, it resolves the generic term “shared secret” to the exact variable KAB (defined in F) to fill the 

[SecretData] slot in S, thereby ensuring the gen-erated security property is accurate and consistent. 

Candidate Deduplication. Finally, to handle mapping complexity, we apply a post-processing step where generated raw properties are parsed into canonical forms. Redundant properties are merged, and any generation that violates the strict type con-straints of the verification tool is filtered out, yield-ing the final set P.

5.2 RQ1: LLM Performance on Goal Extraction 

To answer RQ1, we evaluate representative mod-els across four distinct classes: small-scale open-source models represented by Gemma2-9B-It (Team et al., 2024), large-scale open-source vari-ants such as Qwen3-235B (Yang et al., 2025), proprietary state-of-the-art systems exemplified by GPT-4o (Hurst et al., 2024), and reasoning-optimized specialists including Gemini2.5-Pro (Co-manici et al., 2025) and DeepSeek-V3.2Thinking (Liu et al., 2025). For a granular breakdown and detailed results of security goal extraction perfor-mance, we refer the reader to Appendix E. 

The Precision-Recall Paradox. As illustrated in Figure 3, LLMs exhibit a consistent perfor-mance imbalance across all architectures. While Recall approaches saturation—with reasoning-optimized models like DeepSeek-V3.2Thinking and Gemini2.5-Pro achieving 99.0% and 96.9% respectively—Precision remains the critical bot-tleneck, consistently plateauing below 20% (e.g., 14.3% for Gemini2.5-Pro). This disparity sug-gests that off-the-shelf LLMs function effectively as “security-relevance detectors,” but poorly as “non-security-goal filters.” They can retrieve valid security goal sentences, but drown them out with a large number of non-security goal sentences. Counter-intuitively, enhanced cognitive depth ex-acerbates this issue: reasoning models tend to over-rationalize by interpreting purely operational con-straints (e.g., security measures) as security goals. Consequently, current reasoning heuristics favor broad coverage over the strict boundary delineation required for formal verification.

5.3 RQ2: Effectiveness of SecGoal 

To answer RQ2, we evaluate the impact of domain adaptation on the SecGoal benchmark. We analyze performance from two perspectives: (i) the relative improvement over base models, and (ii) a compara-tive benchmarking against state-of-the-art (SOTA) general-purpose and reasoning-optimized LLMs. 

Direct Effectiveness of SecGoal. As illustrated in Figure 4, incorporating SecGoal into the train-ing pipeline yields remarkable performance gains. Specifically, fine-tuning boosts the F1-score of Qwen2.5-7B-Instruction from 29.4% to 80.4%, and Gemma2-9B-Instruction from 30.3% to 81.7%. Crucially, this improvement is driven by a cor-rection of the precision-recall imbalance: as shown in the blue bars of Figure 4, Precision surges by over 280% across models while Recall remains saturated. This confirms that training on our cu-rated dataset with randomized negative downsam-pling enables models to develop a calibrated de-cision boundary, effectively suppressing the “se-mantic hallucination” where models misclassify operational noise as security goals. 

Benchmarking Against SOTA Models. Our fine-tuned “specialist” models (7B/9B) signifi-cantly outperform substantially larger proprietary architectures. As detailed in Table 1, Gemma2-9B-FT achieves an average F1-score of 81.7%, eclipsing both DeepSeek-V3.2Thinking (29.0%) and Gemini2.5-Flash (26.7%). While our models do not fully match the saturation-level Recall of reasoning-optimized models (94.3% vs. 97.1%), they offer a far superior operational trade-off. The precision gap is decisive: whereas DeepSeek-V3.2Thinking extracts security goals with only 17.7% precision, our models achieve 73.6%. 5.4 RQ3: Formal Property Generation 

To answer RQ3, we evaluate AIFG’s formalization capability through a controlled experiment on Kao Chow v1 and NSSK. We selected these protocols specifically for their tractable complexity, which permits the manual construction of error-free for-mal models as ground truth. This design is cru-cial given the current absence of fully automated tools for extracting formal protocol flows from raw text; it allows us to isolate the framework’s logic mapping performance, effectively decoupling the evaluation from the upstream bottleneck of flow ex-traction. To ensure reliability, we report the mean and standard deviation across three independent runs. The quantitative results, summarized in Ta-ble 2 and further detailed in Appendix F, highlight the framework’s efficacy across three key dimen-sions: 

Completeness and Stability. AIFG demon-strates exceptional stability in security-critical met-rics. We observe perfect Recall ( 1.00 ±0.00 ) across all experimental runs for both protocols. This zero-variance performance is paramount in security analysis; it confirms that the framework is robust against generation stochasticity, ensuring that no intended security properties are omitted from the formal verification process. 

Precise Semantic Grounding. The high and con-sistent slot accuracy ( Acc slot ≈ 0.95 ±0.04 ) sub-stantiates the model’s capability to ground abstract narrative variables into precise protocol symbols. The low standard deviation indicates that our AIFG effectively and reliably bridges the semantic gap between protocol document and security property, minimizing hallucinated variable mappings. 

Conservative Property Synthesis. Regarding extraction precision, the model achieves high av-erage F1-scores ( 0.78 for Kao Chow and 0.83 for NSSK), with peak performance reaching a perfect 

1.0 F1-score in optimal runs. The observed variance in Precision (e.g., 

0.68 ±0.24 ) reflects a tendency towards conservative over-approximation. Qualitative inspection reveals that in runs with lower precision, the model infers plausible but redundant security properties. In for-mal verification, such over-approximation is strictly preferable to under-approximation, as it maximizes scrutiny with negligible computational overhead. 

6 Conclusion 

In this work, we bridge the ”Specification-to-Model Gap” by introducing SecGoal, the first expert-annotated benchmark for security goal extraction, and AIFG, a novel neuro-symbolic framework de-signed to automate this complex task. AIFG distinc-tively decouples the challenge into extraction and formalization, leveraging RAG to ground ambigu-ous natural language into precise formal security properties. This framework effectively resolves the non-bijective mapping between verbose documents and strict formal verification. Our evaluation shows that general-purpose rea-soning is insufficient for filtering the extensive se-curity measures noise in documents. Instead, we demonstrate that instruction tuning on SecGoal en-ables small models to significantly outperform fron-tier models like GPT-4o and Gemini2.5-Pro in secu-rity goal extraction, achieving high recall and preci-sion essential for security-critical domains. While future work must address the trade-off between recall and over-generation, SecGoal and AIFG col-lectively provide the data and methodology to es-tablish AI-assisted security protocol formal verifi-cation as a viable, scalable paradigm for securing next-generation protocols. 87 Limitations 

Bounded by Specification Content. Our frame-work extracts security goals strictly from the source text. However, guaranteeing full protocol secu-rity often necessitates manually supplementing the extracted goals with external security proper-ties derived from domain expertise (Basin et al., 2018). We focus on extracting the specification’s in-tended goals rather than synthesizing these expert-dependent requirements. 

Gap to End-to-End Verification. We target se-curity goal extraction (the “what”) but do not au-tomate the generation of formal protocol models (e.g., protocol flows). As complete formal verifi-cation requires both, our current approach serves as the upstream component. Future work is needed to synthesize executable protocol models directly from raw text to close this loop. 

Scale of Formalization Evaluation. Our evalua-tion of formal property generation (RQ3) focuses on two representative protocols (Kao Chow v1 and NSSK). This scope is necessitated by the aforemen-tioned gap in end-to-end verification: the absence of fully automated tools for extracting formal pro-tocol flows from raw text required us to manually construct and verify ground truth models for the experiments, limiting the feasibility of a large-scale evaluation. 

Generalizability & Scope. While SecGoal cov-ers 15 critical protocols, it may not fully cap-ture the specialized logical structures of emerging domains like Post-Compromise Security (Cohn-Gordon et al., 2016). Furthermore, computational constraints necessitated the exclusion of the ultra-large-scale 5G-AKA protocol from our extraction benchmarks. Finally, we prioritized reproducible, high-performing models (e.g., Gemini2.5-Pro) to establish a solid baseline; consequently, our eval-uation does not exhaustively cover the rapidly ex-panding landscape of contemporary models.

A.1 Hardware and Software Environment 

All fine-tuning experiments were conducted on a computing node based on the Huawei Ascend archi-tecture, equipped with 1 × Ascend NPU (snt9b3). The server features a 24-core ARM processor and 192GB of RAM. The training environment was managed using Python 3.10 and PyTorch 2.1. We utilized the LlamaFactory framework (Zheng et al., 2024) for fine-tuning. For RAG and formalization tasks, we em-ployed the RAGFlow framework (InfiniFlow, 2024). To enhance the quality of property extraction and logical reasoning, we enabled Reasoning Flow and Keyword Analysis features within the pipeline, while keeping all other RAG parameters at their default values. 

A.2 Fine-tuning Methodology 

We employed Low-Rank Adaptation (LoRA) to adapt the instruction-tuned models (Gemma2-9B-It and Qwen2.5-7B-It) to the security property ex-traction task. This approach significantly reduces the number of trainable parameters while main-taining performance. To handle the sparsity of security goals in protocol documents, we imple-mented a randomized negative downsampling strat-egy during training, ensuring a balanced positive-to-negative chunk ratio.

E Qualitative Analysis & Error Examples of Security Goal Extraction 

To dismantle the ”Precision-Recall Paradox” ob-served in RQ1, where frontier models achieve near-perfect recall (often >96%) but suffer from catas-trophic precision (often <15%), we analyze the granular extraction statistics reported in Table 6 and Table 7. The data reveals that this performance gap stems from a fundamental misalignment between general-purpose semantic understanding and the strict re-quirements of formal verification. 

E.1 Quantitative Evidence from Extraction Statistics 

By cross-referencing the extracted sentence count (Ext.), true positives (TP), and property hits (#Hit), we observe distinct failure patterns that explain the low precision: 

The “Needle in a Haystack” Syndrome. High recall is frequently achieved not by precise identifi-cation, but by flooding the output with candidates. This is evident in the analysis of the EDHOC proto-col, where the set of precise security goals is fixed and immutable (GT = 14). To cover this static target, Gemini2.5-Flash extracts a disproportionate 161 sentences (Ext.). While this strategy successfully retrieves most fixed properties (#Hit = 12), the operational cost is se-vere: only 28 of the extracted sentences are valid security goals (TP), leaving 133 false positives (FP). This demonstrates that the model cannot precisely locate the fixed security boundaries; instead, it out-puts nearly 5× more noise than signal, forcing hu-man experts to sift through extensive operational debris to verify the predetermined security goals. 

Reasoning Models Exacerbate Over-Generation. 

Counter-intuitively, reasoning-optimized models often worsen this ratio compared to standard mod-els. On the same EDHOC protocol, DeepSeek-V3.2Thinking extracts 153 sentences, whereas the standard GPT-4o is significantly more conservative, extracting only 61. Despite the massive increase in extraction vol-ume ( +150% ), the reasoning model does not pro-portionally increase valid discoveries; instead, it triples the false positive count (123 FP vs. 40 FP). This indicates that “Chain-of-Thought” prompting encourages models to over-rationalize: they gener-ate justifications for why purely operational steps (e.g., handshake timeouts or message encoding) might be security-relevant, rather than adhering to the strict boundaries of formal analysis. 

E.2 Error Taxonomy 

Based on the statistical behaviors observed above, we categorize the precision failures into two pri-mary cognitive dissonances: 14 E.2.1 Type I: Operational Logic vs. Security Goals 

The observed over-extraction reflects a failure to distinguish security goals (the “what”) from op-erational logic (the “how”). Since specifications overload modal verbs (e.g., “must”, “shall”) to man-date both cryptographic assertions and mechanical interoperability, general-purpose LLMs fall prey to a keyword bias. They treat imperative phrasing as a sufficient condition for security relevance, failing to disambiguate operational constraints from actual protection goals. 

Mechanism Misclassification. Models fre-quently misclassify mechanical constraints, such as parsing rules or compatibility hacks, as accurate security goals. For instance, in TLS 1.3, LLMs erroneously interpret the mandatory check of the legacy version field—a fixed value solely for middlebox interoperability—as a “Downgrade Pro-tection” property. This demonstrates a fundamental failure to distinguish between superficial artifacts required for transmission robustness and the actual cryptographic logic (e.g., the supported versions extension) that guarantees security. 

E.2.2 Type II: The Scope of Formalization 

The behavior of reasoning models—where extrac-tion counts explode (e.g., DeepSeek-Thinking’s 153 extractions on EDHOC) without eliminating errors—points to a lack of domain knowledge re-garding the boundary of formal verification. In symbolic analysis (e.g., Tamarin), security goals must be expressible as logical constraints on traces. 

Out-of-Scope Requirements. Models often ex-tract requirements that are semantically valid but mathematically unverifiable, such as UI mandates (e.g., “display a lock icon”) or physical security constraints. These are vacuously true in symbolic models and should be excluded. 

Over-Inference of Security Goals. Reasoning models are prone to over-inference, frequently hal-lucinating implicit security goals (e.g., Confiden-tiality) from neutral data flows. Unlike GPT-4o, they tend to interpret descriptive transmission steps as prescriptive security claims. This explains the surge in false positives: the models fail to recog-nize that formal verification relies on explicit goals, not inferred intent. 

E.3 Impact of Protocol Complexity and Model Scale 

Further analysis of Table 6 (Small Models) and Table 7 (Large/Reasoning Models) reveals a strong correlation between protocol complexity and per-formance degradation. We observe that the “Precision-Recall Paradox” is not uniform but is significantly exacerbated by the verbosity and struc-tural complexity of industrial documents. 

The Academic-Industrial Gap. Across all model architectures, there is a sharp performance dichotomy between concise academic protocols (e.g., NSSK, Kao-Chow) and monolithic industrial documents (e.g., TLS 1.3, 5G-AKA). • Academic Protocols: On short, structurally dense specifications like NSSK, even small mod-els like Qwen2.5-7B achieve respectable Pre-cision ( ≈31%) and F1-scores ( ≈47%). These documents typically present security logic with minimal operational overhead, making extraction straightforward. • Industrial Standards (Diminishing Returns): 

However, this advantage narrows significantly on notoriously complex industrial documents like TLS 1.3. Here, the immense volume of oper-ational logic saturates the models’ filtering ca-pacity. Gemini2.5-Pro records a Precision of 6.2%, which is statistically comparable to the 6.3% achieved by Gemma2-9B. Despite the larger scale of the advanced model and its superior performance in general inference, it did not significantly outperform the smaller baseline model on some protocol tasks. Both models plateau at a ≈94% False Positive Rate, confirming that without domain-specific alignment (fine-tuning), generic scaling laws yield diminishing returns in filtering the specialized operational noise of indus-trial protocols.
