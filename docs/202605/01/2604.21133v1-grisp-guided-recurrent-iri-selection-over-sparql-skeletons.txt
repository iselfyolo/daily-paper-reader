## Abstract 

We present GRISP (Guided Recurrent IRI Selection over SP ARQL Skeletons), a novel SPARQL-based question-answering method 

over knowledge graphs based on fine-tuning a small language model (SLM). Given a natural-language question, the method first uses the SLM to generate a natural-language SPARQL query skele-ton, and then to re-rank and select knowledge graph items to itera-tively replace the natural-language placeholders using knowledge graph constraints. The SLM is jointly trained on skeleton gen-eration and list-wise re-ranking data generated from standard question-query pairs. We evaluate the method on common Wiki-data and Freebase benchmarks, and achieve better results than other state-of-the-art methods in a comparable setting. 

## Keywords 

SPARQL, Knowledge Graph, Question Answering, SLM, Fine-tuning 

## 1 Introduction 

With RDF 1 gaining more and more popularity to model large sets of data as knowledge graphs, examples being Wikidata [ 18 ], OpenStreetMap [ 4 ], or UniProt [ 10 ], the interest in knowledge graph question-answering (KGQA) methods that generate SPARQL queries, the default query language for RDF data, is growing as well. At the same time, the recent emergence of agentic capabilities in large language models enabled the first zero-shot approaches for SPARQL-based KGQA. Systems like GRASP [ 19 ] and SPINACH [ 11 ]interactively explore and query the knowledge graphs in a human-like fashion, yielding state-of-the-art results on some of the KGQA benchmarks. However, because of their agentic design, they are typically slow and can take minutes to answer queries. Fine-tuning the large underlying model is not feasible without significant hard-ware resources. If question-query pairs are available, methods based on fine-tuning small language models (SLM) can outperform these agentic zero-shot systems, both in generation quality and speed; see [19 ] for an overview. In this work, we present a fine-tuning method using a SLM that operates in a generate-then-retrieve setting and tries to bridge the gap between prior fine-tuning approaches and the iterative retrieval style of agentic systems. 

## Core contributions 

1. We present a novel SPARQL-based knowledge graph question-answering method for arbitrary knowledge graphs, called GRISP. The method is centered around fine-tuning a SLM to perform SPARQL skeleton generation and re-ranking of knowledge graph items to replace the SPARQL skeleton placeholders. 2. We evaluate the method on multiple benchmarks for the Freebase

## 2 Related Work 

There exists a wide body of KGQA methods based on semantic parsing and using fine-tuning, e.g., ChatKBQA [ 12 ], WikiSP [ 21 ], DecAF [ 24 ], and RGR-KBQA [ 8]. We compare against two of these in our evaluation, and describe them in more detail in the following. ChatKBQA [ 12 ] reports state-of-the-art results for semantic-parsing based question-answering methods over Freebase in a generate-then-retrieve setting. ChatKBQA generates S-expression skeletons and later converts them to SPARQL after retrieval for execution, whereas GRISP directly generates SPARQL skeletons. After skeleton generation, ChatKBQA retrieves candidate entities and properties using unsupervised retrieval methods (first inde-pendently for each entity, then independently for each property, but restricting to neighborhoods of previously retrieved entities). In contrast, GRISP performs retrieval iteratively via search and re-ranking, which enables the use of sophisticated knowledge graph constraints to select entities and properties, and allows for back-tracking to recover from selection errors. WikiSP [ 21 ] is a semantic-parsing method in a retrieve-then-generate setting. Contrary to GRISP, it first retrieves entities from the knowledge graph using a fine-tuned version of ReFinED [ 2 ], and generates a SPARQL query using the retrieved entities afterwards. For properties, WikiSP predicts natural-language labels and resolves them using Wikidata APIs. We choose WikiSP because it focuses on the larger and more popular Wikidata knowledge graph, whereas most other works, like ChatKBQA, still evaluate only on Freebase despite it being discontinued since 2016.

IRI-free skeletons. We build on this observation and perform se-mantic parsing first, so we can benefit from the additional guidance provided by the skeletons during retrieval. For example, we use the constraints implied by the skeleton to pre-filter IRI candidates for each placeholder and thereby improve retrieval accuracy. The recurrent selection of IRIs in GRISP is designed after the powerful recurrent retrieval procedure performed by agentic KGQA methods, though restricted to and guided by the generated skeletons. We take the first skeleton that ends up as a fully-resolved SPARQL query that produces a valid and non-empty result as our final answer. Let us look at the core mechanisms of GRISP in the following. 

Beam Search. Beam search has been proven to boost KGQA perfor-mance in the past, both for semantic parsing and knowledge graph traversal [ 1, 12 , 14 ]. We also employ beam search with sampling during the skeleton generation phase to generate multiple SPARQL GRISP: Guided Recurrent IRI Selection over SPARQL Skeletons 

skeleton candidates using the SLM. We filter out duplicate skeletons and those that fail to parse. From the remaining skeletons, we take the top-k according to sequence-log-probability to the selection phase. 

Search and Select. At each natural-language placeholder to be re-solved, we perform an index search with the placeholder label, then re-rank the top-k alternatives using the SLM and select the top-ranked alternative as replacement. Re-ranking is performed by assigning a single-token identifier to each alternative (uppercase letters starting from A in our case), running a single forward pass, and sorting the alternatives according to the respective next-token logit scores of the identifiers, similar to FIRST [ 9]. Depending on the position of the placeholder in the query, we either search an entity index (containing subjects and objects) or a property index, using the search indices provided by [ 19 ]. If the position inference fails, e.g., if a placeholder occurs in a FILTER expression, we search both indices and re-rank the combined set of alternatives. We always add a None alternative to the search results before re-ranking in case no alternative matches and we need to backtrack. When backtracking to a previous state, we re-rank the remaining alternatives that have not yet been selected. 

Backtracking. To allow GRISP to recover from entering a path within the selection tree that leads to a dead end or to an invalid SPARQL query, we allow going back to previous states. For exam-ple, this happens if GRISP re-ranks the None alternative to the top position. To avoid loops, previously selected alternatives in a particular state are filtered out and cannot be selected again. 

Knowledge Graph Guidance. To constrain the set of possible re-placements for the current placeholder, we derive a single-column SPARQL query from the triple patterns connected to the current placeholder in the already resolved prefix of the SPARQL skeleton, execute it, and restrict the corresponding index search to the re-turned candidate IRIs. For example, this allows GRISP to respect constraints such as “Only search within properties known to oc-cur for female politicians born in Canada.” This context-sensitive guidance reduces false positives during selection and often reveals empty or invalid query paths before the skeleton is fully resolved. Note that in its current implementation, placeholders occurring in advanced SPARQL constructs like UNION are unsupported. In such a case we fall back to full-index search. We also add a final query validation step after the SPARQL skeleton is fully resolved. For that, we execute the resolved SPARQL query against the target knowl-edge graph and check that it successfully executes and returns a non-empty result. If these conditions are not met, we backtrack. 

## 4 Evaluation 

Training Data Generation. Given a training sample in the form of a question-query pair, we generate training data for skeleton gen-eration by randomly sampling a question and replacing all IRIs in the SPARQL query with a natural-language placeholder, which is the IRI’s main label 80% of the time, and a random alias otherwise. The definition of main label and alias varies between knowledge graphs, e.g., for Wikidata we use rdfs:label and skos:altLabel ,respectively. For list-wise re-ranking we randomly choose an IRI from the skeleton, then search in the corresponding index with the IRI’s natural-language placeholder to get the selection alternatives for the IRI. To make the model more robust, we perform the fol-lowing data augmentations on the returned selection alternatives independently: a) we randomly vary the search top-𝑘 value between 2 and 10, b) we randomly drop the additional infos added to the alternatives, like schema:description for Wikidata, 20% of the time, c) we randomly drop the target alternative (the one matching the IRI) 20% of the time, and d) we randomly shuffle the alternatives 20% of the time to avoid position bias (most of the time the target alternative is in the top 3 search results). The SLM is jointly trained on both types of data (skeleton gener-ation and list-wise re-ranking) using standard next-token cross-entropy loss. Unlike FIRST, which uses a learning-to-rank loss for re-ranking, we use next-token cross-entropy on the correct iden-tifier, because we only have the target alternative available for supervision, and not a full ordering of all alternatives. Note that the data generation procedure is run for each sample and epoch. This way we present different training data given the same sample to the model in each epoch, which further mitigates overfitting. The data generation can happen either online or offline for multiple epochs in advance. The latter significantly speeds up training and helps with reproducibility. See Figure 1 for an example. 

Training Hyperparameters. We train all models using LoRA ( 𝑟 = 32 ,

𝛼 = 32 ) on all linear layers for 4 epochs with a base learning rate of 

1𝑒 −4, learning-rate warmup and cosine decay, a batch size of 8, and early stopping. We use the Qwen2.5 model family [ 22 ] because it provides a wide range of instruction-tuned models from 0.5B to 7B. We train models up to sizes of 1.5B on a single NVIDIA RTX4090 GPU. For larger models, up to 7B, we train on a single NVIDIA H200 GPU, though smaller GPUs will also work with gradient accumulation and smaller batch sizes. 

Inference Hyperparameters. Unless stated otherwise, we use a tem-perature of 0.4, a top-𝑝 value of 0.9, a beam width of 8, and sampling for skeleton generation. We then keep the top-3 generated skele-tons for the selection phase. For selection, we set the search top-𝑘 

value to 10 , and enable re-ranking, knowledge graph guidance, and backtracking as described above. All models are run on a single NVIDIA RTX4090 GPU. 

Metric. We follow the evaluation protocol of [ 19 ] and evaluate all models and related work using a row-wise F 1-score averaged over samples. We also report the Exact Match (EM) score averaged over samples, which is defined as 1 iff the F 1-score is 1, and 0 otherwise. 

Benchmarks. We evaluate GRISP and compare against ChatKBQA on WebQuestionsSP (WQSP) [ 23 ] and ComplexWebQuestions (CWQ) [ 15 ] for Freebase, and against WikiSP on QALD-7 [ 16 ], WikiWe-bQuestions (WWQ) [ 21 ], SPINACH [ 11 ], and Wikidata Query Logs (WDQL) [ 20 ] for Wikidata. We further evaluate GRISP on QALD-10 [17], LC-QuAD 2.0 [7], SimpleQuestions [6], and QAWiki [13].

On Wikidata in particular, the differ-ences in F 1-score between GRISP and WikiSP are more pronounced, e.g., over 8 p.p. on WWQ dev and over 40 p.p. on WDQL. Note that both ChatKBQA (LLaMa2 7B on WQSP and LLaMa2 13B on CWQ) and WikiSP (LLaMa 7B) use base models which are similar in size to the Qwen2.5 7B model used by GRISP. To provide a fairer comparison on the same base model, we additionally reproduce WikiSP using Qwen2.5 7B (WikiSP 2 in Table 1) and find results similar to the original LLaMa 7B-based results, while still being consistently outperformed by GRISP. We also observe an inverse relationship between F 1-score and average query execution time of GRISP. On more challenging benchmarks with harder questions like SPINACH, GRISP generates more involved SPARQL skeletons, and during retrieval consequently performs more searches, re-ranking, and selection steps, and backtracks more often. Average query times range from 2.3s to 13.8s depending on benchmark complexity, with most of the time being spent on index searches and the final query validation. Note that we find GRISP to be much faster than alter-native agentic methods built on LLMs like GRASP, when we run both of them end-to-end on Wikidata. We further find that GRISP produces a sensible SPARQL query for simple questions (“Where does Barack Obama live”) almost always, for questions of medium difficulty (“Average number of cities with more than 1M inhabitants per country in Europe”) most of the time, and for hard questions (“All British monarchs”) sometimes. Overall, GRISP demonstrates strong performance across the board, and even clearly outperforms GRASP on some benchmarks. For example, GRASP with Qwen2.5 72B achieves an F 1-score of 40.8 on WQSP and 68.7 on WWQ, com-pared to GRISP’s 76.0 and 79.5. However, if there is relatively little training data available, like for QALD-7 (100 pairs) or QALD-10 (412 pairs), zero-shot agentic methods take the upper hand. On these two benchmarks, GRASP with Qwen2.5 72B achieves an F 1-score of 73.5 and 67.0 compared to GRISP’s 46.0 and 47.6. But even in this low-data regime, GRISP can still be a good choice depending on the desired tradeoff between answer quality, inference speed, and the ability to run on a single consumer GPU. 

Component and Ablation Studies We also look at the effect of different components of GRISP on its question-answering perfor-mance. All studies are carried out on the development set of WWQ with a GRISP model trained on the WWQ training set. The results are shown in Table 3. We first perform studies on two important inference hyperpa-rameters, which are the beam width for skeleton generation and the top-𝑘 value for search. As expected, we find that for both param-eters, performance of GRISP improves with increasing parameter values. But we also quickly hit diminishing returns for both param-eters: a beam width larger than 8 or a search top-𝑘 value larger than 10 do not lead to further improvement. We therefore set the default beam width and search top-𝑘 value of GRISP in our code to 8 and 10, respectively. When altering the size of GRISP’s underlying SLM, we get con-sistent performance improvements with larger models. The im-provements are significant, both from 0.5B to 1.5B and from 3B to 7B. But even the 0.5B model reaches the Qwen-based WikiSP score of 74.8 on WWQ dev , which the 1.5B then clearly surpasses. Lastly, we check the effect of removing core mechanisms from GRISP. We find that all mechanisms add value to the GRISP method, but the largest drop in performance is caused by removing beam search, followed by re-ranking, knowledge graph guidance (con-straining selection and validating final SPARQL query), and lastly backtracking. We suspect that the latter two would have a bigger impact on performance if GRISP is run on a knowledge graph with a more complex schema than Wikidata, or on more challenging questions than WWQ dev offers. 

## 5 Conclusion 

We present GRISP, a novel KGQA method in a generate-then-retrieve setting based on fine-tuning a SLM on the tasks of SPARQL skeleton generation and list-wise re-ranking of knowledge graph items. The main innovation of GRISP is the retrieval stage, where we iteratively replace natural-language placeholders in SPARQL skeletons with IRIs via search and list-wise re-ranking. This re-current procedure also allows for sophisticated knowledge graph constraints and backtracking during inference. We evaluate GRISP against state-of-the-art methods that operate in a comparable set-ting and achieve consistent improvements on benchmarks for Free-base and Wikidata. Notably, GRISP is much faster than current agentic methods but can match and even surpass their performance given sufficient training data, all while running on a single con-sumer GPU.
