## 摘要
使用合成推理数据训练大语言模型（LLMs）已成为增强其推理能力的流行方法，而影响该范式有效性的关键因素是生成的逐步推理数据的质量。为了生成高质量的推理数据，近期许多方法生成合成推理路径并根据最终答案的正确性进行过滤，这往往忽略了中间推理步骤中的缺陷。为了加强对中间推理步骤的验证，先前的工作主要求助于代码执行或符号推理引擎。然而，基于代码的验证仅限于代码或数学任务，而推理引擎则需要结构良好且完整的上下文。因此，现有方法在涉及模糊或不完整上下文的自然语言推理任务中无法有效发挥作用。在这些任务中，合成数据仍然缺乏验证每个推理步骤的可靠检查。为了应对这一挑战，我们引入了 ORACLE，这是一个受三段论推理启发的结构化数据生成框架。ORACLE 将 LLM 的生成优势与符号监督相结合：LLM 生成逐步推理上下文，而符号推理引擎验证每个中间步骤的有效性。通过采用统一的提示模板来诱导模块化推理链，ORACLE 实现了细粒度的步骤级验证，从而促进了高质量多步推理数据的构建。在六个逻辑、事实和常识推理基准测试中，我们的 ORACLE 在多个模型上始终优于强基准模型。

## Abstract
Training large language models (LLMs) with synthetic reasoning data has become a popular approach to enhancing their reasoning capabilities, while a key factor influencing the effectiveness of this paradigm is the quality of the generated multi-step reasoning data. To generate high-quality reasoning data, many recent methods generate synthetic reasoning paths and filter them based on final answer correctness, often overlooking flaws in intermediate reasoning steps. To enhance the verification of intermediate reasoning steps, prior work primarily resorts to code execution or symbolic reasoning engines. However, code-based validation is restricted to code or mathematical tasks, and reasoning engines require a well-structured and complete context. As a result, existing methods fail to function effectively in natural language reasoning tasks that involve ambiguous or incomplete contexts. In these tasks, synthetic data still lack reliable checks for verifying each reasoning step. To address this challenge, we introduce ORACLE, a structured data generation framework inspired by syllogistic reasoning. ORACLE integrates the generative strengths of LLMs with symbolic supervision: the LLM produces step-wise reasoning contexts, while a symbolic reasoning engine verifies the validity of each intermediate step. By employing a unified prompting template to elicit modular reasoning chains, ORACLE enables fine-grained, step-level validation, facilitating the construction of high-quality multi-step reasoning data. Across six logical, factual, and commonsense reasoning benchmarks, our ORACLE consistently outperforms strong baselines on multiple models.

论文的核心问题在于**如何提升合成推理数据的质量，以有效增强大语言模型（LLM）的复杂推理能力**。
*   **研究动机**：目前主流方法（如RFT、STaR）主要依赖“最终答案正确性”来过滤合成数据。这种“结果导向”的筛选容易引入“捷径学习”（Shortcut Learning），即模型可能通过错误的逻辑步骤凑出正确答案。
*   **背景挑战**：虽然代码执行或符号引擎可以验证中间步骤，但代码验证局限于数学/编程，而符号引擎对自然语言中模糊、不完整的上下文适应性较差。因此，在自然语言推理任务中，缺乏一种可靠的、细粒度的中间过程验证手段。

### 2. 方法论：核心思想与关键技术细节
ORACLE 提出了一种受**三段论（Syllogistic Reasoning）**启发的结构化数据生成框架，核心思想是将 LLM 的生成能力与符号推理引擎的严谨性相结合。
*   **结构化模板**：设计了统一的响应模板，包含 `<QUERY>`（子问题）、`<FACTS>`（相关事实）、`<RULE>`（推理规则）、`<REVISION>`（自我修正）和 `<REASONING_RESULT>`（推理结果）。这种模块化设计便于符号引擎提取和验证。
*   **两阶段训练流程**：
    1.  **阶段一：格式内化**。利用少量人工示例通过 Few-shot 诱导模型生成初步数据，筛选出格式正确且答案正确的样本对基座模型进行微调（SFT），使其掌握结构化推理格式。
    2.  **阶段二：符号监督增强**。
        *   **Beam Search 引导生成**：在生成每一步时，LLM 生成自然语言步骤，随后由模型翻译为符号语言，交由符号推理引擎（如 Pyke）执行。
        *   **综合评分机制**：根据引擎执行是否成功（W1）、LLM 对准确性的评估（W2）以及可行性评估（W3）进行综合打分，保留高质量路径。
        *   **数据产出**：将验证通过的完整路径作为 SFT 数据；将验证通过的节点与其失败的兄弟节点组成偏好对，进行直接偏好优化（DPO）。

### 3. 实验设计
*   **数据集**：涵盖六大基准测试，包括逻辑推理（ProntoQA, ProofWriter）、事实验证（BoolQ, StrategyQA）、常识推理（CosmosQA）和科学多跳推理（ScienceQA）。
*   **对比方法**：
    *   基础 CoT（Zero-shot / Four-shot）。
    *   RFT（拒绝采样筛选）。
    *   ToT-SFT（基于树搜索的微调）。
    *   Self-rewarding（模型自评分 DPO）。
*   **实验模型**：Llama-3.1-8B-Instruct, Mistral-7B-Instruct-v0.3, Qwen-2.5-7B-Instruct。

*   **训练细节**：采用 LoRA（rank=8）高效微调。SFT 阶段使用 12k 样本，DPO 阶段使用 2k 偏好对。

*   **实验规模**：在 3 个主流模型和 6 个不同类型的推理数据集上进行了交叉验证，总计进行了数十组对比实验。
*   **消融实验**：分别验证了符号引擎、Beam Search 和 DPO 模块的贡献，证明了各组件的必要性。
*   **分析深度**：额外分析了推理引擎的执行成功率以及失败案例的错误类型（如生成错误 vs 翻译错误）。
*   **评价**：实验设计覆盖了从符号逻辑到常识推理的多种场景，对比基准涵盖了当前主流的合成数据增强技术，实验结果具有较强的说服力和客观性。

### 6. 主要结论与发现
*   **性能领先**：ORACLE 在所有测试的数据集和模型上均显著优于强基准方法，尤其在逻辑严密性要求高的任务（如 ProofWriter）中提升幅度最大。
*   **验证有效性**：引入符号引擎能显著减少逻辑不一致的路径，提升中间步骤的忠实度。
*   **迭代增益**：随着训练迭代，模型生成标准化、可验证推理步骤的能力会自我进化（成功率提升）。

### 7. 优点
*   **细粒度监督**：通过符号引擎实现了对自然语言推理“过程”的硬约束，而非仅仅关注“结果”。
*   **可解释性强**：模块化的模板设计使推理过程透明，便于自动分析和人工纠错。
*   **通用性扩展**：成功将符号逻辑验证引入到了非结构化的常识和科学推理领域。

### 8. 不足与局限
*   **翻译瓶颈**：框架高度依赖 LLM 将自然语言准确翻译为符号语言的能力，翻译错误（Translation Error）是导致验证失败的主要原因之一。
*   **计算开销**：Beam Search 结合符号引擎的生成过程比传统的采样方法更耗费推理算力和时间。
*   **复杂场景限制**：在处理极度模糊或高度依赖隐性常识的推理时，符号化表达依然存在挑战，导致部分任务的引擎执行成功率较低。
