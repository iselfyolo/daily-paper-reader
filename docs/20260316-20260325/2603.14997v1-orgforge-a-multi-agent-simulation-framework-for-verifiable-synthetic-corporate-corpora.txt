Abstract 

Evaluating retrieval-augmented generation (RAG) pipelines requires corpora where ground truth is knowable, temporally structured, and cross-artifact—properties that real-world datasets rarely provide cleanly. Existing resources such as the Enron corpus carry legal ambiguity, demographic skew, and no structured ground truth. Purely LLM-generated synthetic data solves the legal problem but introduces a subtler one: the generating model cannot be prevented from hallucinating facts that contradict themselves across documents. We present OrgForge , an open-source multi-agent simulation framework that enforces a strict physics-cognition boundary : a deterministic Python engine maintains a SimEvent ground truth bus; large language models generate only surface prose, constrained by validated proposals. An actor-local clock enforces causal timestamp correctness across all artifact types, eliminating the class of timeline inconsistencies that arise when timestamps are sampled independently per document. We formalize three graph-dynamic subsystems—stress propagation via betweenness centrality, temporal edge-weight decay, and Dijkstra escalation routing—that govern organiza-tional behavior independently of any LLM. Running a configurable N -day simulation, OrgForge produces interleaved Slack threads, JIRA tickets, Confluence pages, Git pull requests, emails, and server logs, all traceable to a shared, immutable event log. We additionally describe a causal chain tracking subsystem that accumulates cross-artifact evidence graphs per incident, a hybrid reciprocal-rank-fusion recurrence detector for identifying repeated failure classes, and an inbound/outbound email engine that routes vendor alerts, customer complaints, and HR correspondence through gated causal chains with probabilistic drop simulation. OrgForge is available under the MIT license.

## 1. Introduction 

Retrieval-augmented generation has become a primary deployment pattern for enterprise LLM applications, yet the evaluation infrastructure lags the deployment reality. Benchmarks for RAG typically test either retrieval quality in isolation or end-to-end question answering on static Wikipedia-style passages. Neither captures the properties of real enterprise knowledge bases: documents that reference each other across systems, facts that evolve over time, incidents that leave traces in multiple artifact types simultaneously. The Enron corpus (Klimt & Yang, 2004) remains the most widely used real corporate dataset for NLP research, despite being two decades old, legally fraught, and representing a single pathological organizational context. More recent alternatives exist—FinanceBench, MultiHop-RAG (Tang & Yang, 2024), LongBench (Bai et al., 2023)—but none provide the cross-system, temporally grounded, ground-truth-annotated structure that enterprise RAG evaluation requires. The tempting solution is to generate synthetic data with large language models. LLMs write convincing Slack messages and JIRA tickets. The problem is that LLM-generated corpora have no external ground truth to validate against. If a language model generates a Slack thread in which an engineer says “the auth service has been down since 3am” and separately generates a JIRA ticket that records the incident as starting at 9am, there is no mechanism to detect or prevent the contradiction. The corpus looks realistic but is internally inconsistent in ways that will silently corrupt any RAG evaluation run against it. OrgForge addresses this through a strict architectural boundary: LLMs propose, the engine executes. 

A deterministic simulation engine controls all facts—who is on call, what time an incident started, which engineer owns a ticket, what the current system health is. Language models receive validated context and generate surface prose only. Every significant action emits a structured SimEvent to a persistent log. The corpus and the ground truth bus are produced by the same simulation run, making them structurally guaranteed to be consistent. This paper makes the following contributions: 

• A simulation architecture formalized as M = ( S, P, V, E ) that separates fact control from prose generation, preventing LLM hallucinations from contaminating synthetic corpora. 

• Three formally specified graph-dynamic mechanisms—stress propagation, edge decay, and Dijkstra escalation—that govern organizational behavior deterministically. 

• A proposal-validation loop that allows LLM planners to influence narrative direction without bypassing factual constraints. 

• A multi-artifact output schema grounded in a shared SimEvent bus, enabling cross-artifact reasoning benchmarks. 

• An actor-local clock ( sim clock.py ) that guarantees causal timestamp correctness across all artifact types, eliminating the class of timeline inconsistencies produced by independent per-artifact timestamp sampling. 

• An ambient social interruption model that introduces ground-truth capacity noise into the corpus, enabling evaluation of retrieval systems against realistic benign communication rather than purely task-driven artifacts.

## 2. Background and Motivation 

2.1 Limitations of Existing Corpora 

The Enron email dataset (Klimt & Yang, 2004) has been used in hundreds of NLP studies but carries well-documented limitations: it represents a single company in crisis, skews heavily toward senior management communication, and is subject to ongoing legal sensitivity. More practically, it contains no structured ground truth—there is no authoritative record of what actually happened that researchers can use to verify whether a RAG system retrieved the correct facts. Synthetic alternatives generated by LLMs have proliferated with the availability of capable models, but they inherit the hallucination problem at the corpus level. When the generating model has no external constraint on factual consistency, the resulting corpus cannot be used as a reliable evaluation benchmark. A system that retrieves an internally inconsistent document and returns a “correct” answer by coincidence is indistinguishable from one that retrieves and reasons correctly. 

2.2 RAG Evaluation Requirements 

Effective RAG evaluation requires corpora with at least four properties that existing resources lack simultaneously: 1. Traceable ground truth: each retrievable fact must have a canonical source that can be used to score retrieval. 2. Temporal structure: facts must change over time so that temporal reasoning—“what was the state of X on day N ?”—can be evaluated. 3. Cross-artifact coherence: the same fact must appear consistently across multiple document types, enabling multi-hop retrieval evaluation. 4. Configurable complexity: incident severity, organizational size, and communication patterns should be tunable to probe specific system behaviors. OrgForge is designed to satisfy all four.

2.3 Related Work 

MultiHop-RAG (Tang & Yang, 2024) focuses on multi-hop reasoning but uses news articles without organizational structure. FRAMES (Krishna et al., 2024) provides factual, temporal, and multi-document questions but is not grounded in a controllable simulation. Enterprise RAG benchmarks accompanying proprietary systems are not publicly available for independent evaluation. In the agent-based simulation literature, social network simulations have been used to study information diffusion (Watts & Strogatz, 1998) and epidemic spread, but have not been connected to document corpus generation for NLP evaluation. OrgForge bridges this gap by using the organizational simulation as the ground truth engine and the document corpus as the derived artifact. 

## 3. System Architecture 

OrgForge runs a discrete-time simulation over N days. Each day proceeds through a planning phase , an execution phase , and an end-of-day summarization phase . The core invariant maintained throughout is that the SimEvent log is the sole authoritative record of facts; all generated text is prose grounded in that record. The execution flow follows a fixed pipeline: graph dynamics update the organizational state; the validator screens LLM proposals against that state; the planner orchestrates multi-department agendas; artifact generators dispatch prose to each channel; and the causal chain handler and email engine extend that artifact graph beyond the organization boundary. The subsections below follow this execution order. 

3.1 Formal System Definition 

We define the OrgForge simulation as the tuple M = ( S, P, V, E ), where: 

• S (State): A Pydantic model representing all mutable simulation variables. Relevant fields include system health H ∈ [0 , 100], team morale μ ∈ [0 , 1], active incidents, open JIRA tickets, Confluence page registry, and per-engineer stress scores. 

• P (Planners): A set of LLM-based department agents that observe S and the SimEvent history, then generate structured JSON proposals for the day’s activities. Planners influence narrative direction but cannot mutate S or write to E directly. 

• V (Validator): A deterministic function V : ProposedEvent × S × E → { 0, 1} that admits or rejects each proposal before execution. Described in full in Section 3.5. 

• E (Events): The SimEvent log—a persistent, append-only record of every significant action. This is the ground truth bus. LLMs retrieve context from E via hybrid vector search but never write to it. The boundary V separates the “physics” layer ( S, E, graph dynamics) from the “cognition” layer (P ). This is the property that prevents hallucinations from contaminating the corpus.

3.9 Causal Chain Tracking and Recurrence Detection 

Artifact generation does not terminate when the day plan is dispatched. Once an incident opens, OrgForge instantiates a CausalChainHandler that accumulates an ordered list of artifact IDs as the incident progresses— the initial Slack bot alert, the JIRA ticket, PR merges, postmortem Confluence pages, and any external email threads that surface the same root cause. Each downstream artifact produced by the generators in Section 3.8 is appended to the chain in chronological order. Snapshots are written into each SimEvent’s facts payload, so the chain as it existed at every moment is recoverable without post-hoc reconstruction.

3.10 External Email Engine 

The email engine extends the causal chain system across the organization boundary. OrgForge generates three categories of external email during each simulation day; every category feeds back into the same CausalChainHandler and SimEvent log that govern internal artifacts. 

Tech vendor inbound. Automated alerts from vendors (AWS quota warnings, Snyk security findings, Stripe billing notices) arrive pre-standup (06:00–08:59). The recipient is not the department lead by default; instead, ExternalEmailIngestor uses Memory.find expert by skill() to route each alert to the engineer whose expertise embedding has the highest cosine similarity to the alert topic. If the alert topic overlaps with an active incident’s root cause, the email is automatically appended to that incident’s CausalChainHandler , extending the evidence graph that the evaluation harness later traverses. 

Customer complaint inbound. Customer emails arrive during business hours (09:00–16:30) and traverse a gated routing chain: Sales receives a Slack ping, Product makes a triage decision, and high-priority complaints ( p = 0 .55) generate a JIRA ticket. Approximately 15% of customer emails are probabilistically dropped—no downstream action is taken. These are logged as email dropped 

SimEvents with no downstream artifacts, creating verifiable communication gaps that the evaluation harness targets with GAP DETECTION questions. 

HR outbound. Offer letters and onboarding preparation emails are fired 1–3 days before each scheduled hire’s arrival day. The email artifact is linked into the employee hired causal chain on arrival, so an evaluation agent can trace the complete pre-hire communication sequence. 

Source generation. External email sources (vendor names, customer contacts, email addresses, routing department, trigger conditions, topic pools) are generated once by LLM during the simulation genesis phase, grounded in the company’s established tech stack, and persisted to MongoDB. No static configuration entries are required; changing the industry field automatically shifts the vendor and customer ecosystem. 

3.11 Memory and Retrieval Layer 

The execution pipeline uses a vector memory layer throughout: planners retrieve context, the email router matches alerts to experts, and the recurrence detector searches prior incidents. Two components of this layer are worth describing explicitly. 

HyDE retrieval. The Memory.recall with rewrite() method implements Hypothetical Docu-ment Embeddings (HyDE): before embedding a retrieval query, an LLM generates a hypothetical passage matching the query, and that passage is embedded instead of the raw query. This substan-tially improves recall for ad-hoc documentation and design-discussion topics whose vocabulary differs from the query terms—a common failure mode when planners retrieve context for low-frequency event types. 

Persona skill embeddings. At genesis, each employee’s expertise tags are embedded and stored as persona skill artifacts in the vector store. find expert by skill() performs semantic lookup over these embeddings, enabling expertise-aware routing for both email delivery (Section 3.10) and Slack thread participant selection without relying on explicit department membership rules.

## 4. Corpus Properties 

4.1 Cross-Artifact Traceability 

A single incident in OrgForge produces a traceable chain of artifacts: a server log anomaly triggers an incident opened SimEvent; the on-call engineer posts a Slack alert; a JIRA incident ticket is created with the SimEvent ID in its payload; external contacts are engaged; a postmortem Confluence page is drafted; a GitHub PR fixing the root cause is merged. Every artifact in this chain references the same incident ID and is appended to the incident’s CausalChainHandler in chronological order (Section 3.9). A RAG pipeline evaluated against this corpus can be tested on whether it can reconstruct the full incident timeline from any single entry point. External email threads extend this traceability beyond the organization boundary. A vendor alert that overlaps an active incident’s root cause is appended to the same causal chain as the internal Slack threads and JIRA ticket, so a complete chain may span inbound email, internal coordination artifacts, and outbound postmortem—all with monotonically non-decreasing timestamps and a shared chain identifier. 

4.2 Temporal Structure 

The N -day simulation produces facts with known temporal validity windows. System health Ht

degrades on incident opening and recovers on resolution. Tickets open, progress, and close. Edge weights wuv,t evolve continuously via Eq. (4) . Engineer stress Si,t rises on incidents and falls via daily recovery in Eq. (3) . This temporal structure enables evaluation of questions such as “which incidents were open on day 6?” that require temporal reasoning rather than simple lookup. 

4.3 Causal Timestamp Consistency 

Early versions of OrgForge assigned artifact timestamps via independent random.randint calls per artifact type. This produced causally inconsistent timelines: a PagerDuty alert could carry a timestamp later than the Slack thread responding to it, and a single engineer could appear to perform two concurrent activities at the same wall-clock time. We address this with sim clock.py , an actor-local clock in which every employee maintains an independent time cursor. Timestamps are no longer sampled independently—they are derived from cursor state via two core primitives: 

• advance actor( i, ∆) — models ambient parallel work (ticket progress, Confluence edits, deep work). Advances only actor i’s cursor by ∆ and samples the artifact timestamp from within that work block. All other actors are unaffected, preserving genuine parallelism in the workday. 

• sync and tick( A, ∆) — models causal chains (incident response, Slack threads, PR reviews). Synchronises all participants in actor set A to max i∈A( cursor i), then advances the shared cursor by ∆. This guarantees that no response artifact can receive a timestamp earlier than its trigger.

4.4 Ambient Social Interruption and Capacity Noise 

Real engineering corpora contain benign noise: engineers are pulled into watercooler conversations, non-blocking Slack threads, and informal knowledge-sharing exchanges that consume time without producing primary work artifacts. Purely task-driven synthetic corpora lack this property, which creates an evaluation blind spot—a RAG system tested against noise-free data may fail silently when deployed against realistic enterprise communications. OrgForge models this through a social interruption mechanism. On any simulation day, employees may be drawn into ambient Slack activity (watercooler channels, off-topic threads, informal standups) according to a configurable interruption probability. Each interruption consumes a sampled time quantum from the actor’s clock via advance actor() , reducing the remaining capacity Ci (Eq. (8) )available for planned agenda items. This has two corpus-level effects. First, it produces low-signal artifacts that a retrieval system must learn to discount: a Slack message about lunch plans appearing in the same channel as an incident thread provides a realistic negative example for retrieval evaluation. Second, it introduces ground-truth capacity variance: the SimEvent log records the time consumed by interruptions, so benchmark questions of the form “why did ticket X take three days longer than estimated?” have verifiable answers grounded in the interruption log rather than arbitrary narrative choices. The interruption model deliberately generates benign noise. This addresses a recognized gap in synthetic evaluation data: the preponderance of adversarial or crisis-driven corpora means that models trained or evaluated on them may over-index on high-signal events and underperform on the routine, low-stakes communication that comprises the majority of real enterprise knowledge bases. 

4.5 Configurable Organizational Complexity 

OrgForge’s config.yaml specifies company name, organizational structure, persona definitions (name, role, communication style, expertise), and incident triggers. Teams of 5 to 50+ can be simulated. Incident frequency, external contact probability, and ad-hoc document generation rates are all configurable. Corpora are reproducible by fixing the random seed.

5.1 Causal Thread Extraction 

CausalThreadBuilder reads the SimEvent log and reconstructs four classes of thread: 

• Incident threads — rooted at incident opened , spanning Slack alerts, JIRA tickets, PRs, and postmortems. Each node is annotated with known to : the cumulative set of actors who had encountered the artifact at that timestamp, enabling temporal knowledge-state queries. 

• Customer email threads — rooted at inbound external email , spanning the Sales Slack ping, Product triage decision, and optional JIRA ticket. 

• Dropped email threads — single-node chains for emails that received no downstream action (email dropped SimEvents). These are intentional ground-truth gaps: a correct agent should detect the absence of any downstream artifact. 

• HR threads — rooted at hr outbound email , linked to the corresponding employee hired 

event on the hire’s arrival day. 

5.2 Question Generation 

EvalQuestionGenerator produces eight typed question categories from the extracted threads and SimEvent log: 1. RETRIEVAL — “Which artifact first documented root cause X?” Ground truth: the root artifact ID and timestamp. 2. CAUSAL — “What artifact was produced as a direct result of Y ?” Ground truth: next-node artifact ID and event type; requires both to match for full credit. 3. TEMPORAL — “Did person P have access to documentation about domain D when incident 

I opened?” Ground truth: a boolean derived from whether a employee departed event for D’s knowledge owner precedes the incident day. 4. GAP DETECTION — “Was the email from source S ever actioned?” Ground truth: boolean 

was actioned derived from the presence or absence of downstream artifacts in the causal chain. 5. ROUTING — “Who was the first internal person to receive the complaint from S?” Ground truth: the liaison name from the inbound external email SimEvent. 6. PLAN — “Which department was planning around theme T on day D?” Ground truth: derived from dept plan created SimEvents. 7. ESCALATION — “Who was pulled into the escalation chain for incident I?” Ground truth: the ordered actor list from the escalation chain SimEvent, including hop count. 8. KNOWLEDGE GAP — “Which domain was undocumented when incident I fired?” Ground truth: the gap areas list from the knowledge gap detected SimEvent; partial credit awarded proportionally to domain overlap.

Each question record includes a difficulty label (easy / medium / hard), a requires reasoning 

flag, and an evidence chain listing the artifact IDs that support the ground-truth answer. The evidence chain enables partial-credit scoring of retrieval quality independently of answer correctness. 

5.3 Deterministic Scoring 

OrgForgeScorer dispatches each question to a type-specific scorer. The final score is a weighted combination of primary answer correctness and evidence retrieval quality: score( q, a ) = min(1 .0, 0.80 · sprimary (q, a ) + 0 .20 · sevidence (q, a )) (11) where sprimary ∈ [0 , 1] measures answer correctness and sevidence is the Jaccard overlap between the ground-truth evidence chain and the artifact IDs the agent reports retrieving. The evidence weight is deliberately secondary (0.20) so that a lucky retrieval cannot mask a wrong answer. Type-specific primary scoring rules include: 

• RETRIEVAL : exact artifact ID match (1.0); timestamp within 30 minutes of ground truth adds a +0 .10 proximity bonus. 

• CAUSAL : requires artifact ID match and event type match for full credit (1.0); artifact match alone yields 0.5 primary. 

• TEMPORAL : exact boolean match (1.0); departure-day agreement within one day adds a +0 .10 bonus. 

• GAP DETECTION : boolean was actioned match (0.6 primary) combined with downstream artifact overlap (0.4 primary). 

• KNOWLEDGE GAP : Jaccard overlap over gap areas sets, scaled from 0.4 (partial) to 1.0 (exact match). A question is marked correct when score ≥ 0.90 and partial when 0 .20 ≤ score < 0.90. 

OrgForgeScorer.report() aggregates results by question type, difficulty, and reasoning complexity (direct lookup vs. multi-hop traversal).

5.6 Ongoing and Future Experiments 

One experiment is currently in progress; two remain planned for a subsequent version of this paper. 

Experiment 1 — End-to-end RAG agent evaluation. Apply a full RAG pipeline (retrieval + generation) to the benchmark and report weighted OrgForgeScorer scores broken down by question type, difficulty label, and requires reasoning flag. The retrieval baselines above establish the evidence-surfacing ceiling; this experiment measures how well a generation layer converts retrieved evidence into correct answers. 

Experiment 2 — Cross-corpus contradiction rate (in progress) . A second corpus is currently being generated using the same LLM without the SimEvent constraint (unrestricted generation, same prompt templates). For each matched document pair referencing the same incident, we will measure the per-field contradiction rate on timestamp, actor, and ticket-status fields. The expected result is that the unconstrained corpus shows contradiction rates significantly above zero while the OrgForge corpus shows zero by construction, directly operationalizing the core architectural claim. Results will be reported in a subsequent version of this paper. 

Experiment 3 — Temporal and gap-detection sensitivity. Evaluate TEMPORAL and GAP DETECTION questions with and without SimEvent metadata provided as retrieval context, measuring whether direct access to the ground-truth event log improves answer correctness. Ad-ditionally, measure what fraction of probabilistically dropped emails are correctly identified as unactioned by a retrieval agent.

## 7. Limitations and Future Work 

7.1 Current Limitations 

• Persona diversity. Communication styles are currently hand-authored in config.yaml . Auto-matic persona generation from demographic distributions would improve coverage. 

• LLM dependency. Artifact prose quality depends on the underlying model. Factual consistency is guaranteed by the SimEvent architecture regardless, but weaker models produce less realistic surface text. 

• Partial empirical validation. BM25 and dense retrieval baselines have been collected against a 22-business-day corpus (1,079 documents; Section 5.5). End-to-end RAG agent evaluation, cross-corpus contradiction measurement, and temporal/gap-detection sensitivity analysis remain outstanding and are described as future experiments in Section 5.5. 

• Single-company scope. OrgForge simulates one organization. The external email engine introduces vendor and customer contacts, but they are passive sources—generated inbound messages, not independent organizations with their own state machines, graph dynamics, or planning loops. Full vendor–customer cross-organization simulations, where both sides maintain verifiable state, are not yet supported. The email engine is a partial step in that direction. 

7.2 Future Work Multi-company simulations. Extending the external contact model to support full vendor– customer cross-organization simulations would enable evaluation of inter-company knowledge hand-offs and supply-chain incident propagation. 

Historical institutional corpora. The OrgForge simulation model applies naturally to historical administrative records—classical antiquity bureaucratic correspondence, medieval guild records,

early modern trading company documents—where organizational structure and communication patterns are known but corpora are sparse. Collaboration with digital humanities researchers to generate ground-truth-annotated synthetic historical corpora would open evaluation opportunities across cross-disciplinary venues. 

Community event backlog. Novel events logged via drain novel log() represent the LLM’s latent knowledge of organizational activities not yet in the schema. Systematic review of these proposals will drive future KNOWN EVENT TYPES extensions. 

Plugin architecture. A formal plugin interface for community artifact types (Zoom transcripts, Zendesk tickets, PagerDuty timelines) is on the roadmap, enabling the corpus to cover the full surface area of enterprise communication without changes to the core simulation engine. 

## 8. Conclusion 

We have presented OrgForge, a multi-agent simulation framework for generating synthetic corporate corpora with verifiable ground truth. The central contribution is an architectural boundary formalized as M = ( S, P, V, E ) that separates fact control from prose generation. We have formally specified three graph-dynamic mechanisms—stress propagation via betweenness centrality (Eq. (3) ), temporal edge-weight decay (Eq. (4) ), and Dijkstra escalation routing (Eq. (7) )—that govern organizational behavior deterministically and independently of any LLM. An actor-local clock enforces causal timestamp correctness across all artifact types, and an ambient social interruption model introduces ground-truth capacity noise representative of realistic enterprise communication. Beyond the core simulation, we have described a causal chain tracking subsystem that accumulates ordered artifact graphs per incident, a hybrid RRF recurrence detector that identifies repeated failure classes across the SimEvent log, and an inbound/outbound email engine that introduces three categories of external communication with probabilistic drop simulation and gated causal chains. These components are integrated into the core execution pipeline—the email engine and causal chain handler extend the same artifact flow driven by the validator and planner, rather than operating as post-processing additions. A complete evaluation pipeline generates eight typed question categories with deterministic ground-truth answers, scores agent responses with partial credit via a weighted primary-evidence scheme, and exports HuggingFace-ready Parquet datasets with BM25 and dense retrieval baselines. The resulting corpora are internally consistent by construction, temporally structured, causally ordered, and cross-artifact traceable. BM25 and dense retrieval baselines have been evaluated against a 22-business-day corpus (1,079 documents, 83 questions, 43 simulated personnel); BM25 outperforms dense retrieval on most question types, with dense narrowly leading on RETRIEVAL (MRR@10 0.39 vs. 0.35), and PLAN and ESCALATION scoring zero for both methods, establishing a clear target for future agentic systems. End-to-end RAG agent evaluation against these corpora remains the primary outstanding item and will be reported in a subsequent version of this paper. OrgForge is open source, configurable, and reproducible. We believe it provides the foundation for a new class of RAG evaluation benchmarks grounded in organizationally realistic, ground-truth-annotated synthetic corpora.
