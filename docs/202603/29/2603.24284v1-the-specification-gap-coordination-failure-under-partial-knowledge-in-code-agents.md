## 摘要
当多个基于大语言模型（LLM）的代码智能体独立实现同一个类的不同部分时，即使规范说明（specification）中未明确给出这些选择，它们也必须在共享的内部表示上达成一致。我们通过 51 个类生成任务研究了这一协作问题，逐步剥离规范细节（从完整的文档字符串 L0 到仅剩函数签名 L3），并引入相反的结构偏差（列表与字典）以对集成进行压力测试。研究得出三个发现：首先，存在持久的规范差距，随着细节的移除，双智能体集成准确率从 58% 下降到 25%，而单智能体基准的下降则更为平缓（从 89% 到 56%），从而留下了 25-39 个百分点（pp）的协作差距，这一结果在两个 Claude 模型（Sonnet、Haiku）和三次独立运行中保持一致。其次，一个基于抽象语法树（AST）的冲突检测器在最弱的规范级别下达到了 97% 的精确度，且无需额外的 LLM 调用；然而，析因恢复实验表明，仅恢复完整的规范说明就能恢复到单智能体的性能上限（89%），而提供冲突报告并没有带来可衡量的收益。第三，将差距分解为协作成本（+16 pp）和信息不对称（+11 pp）表明，这两种效应是独立的且具有近似可加性。这种差距不仅仅是隐藏信息的结果，还反映了在没有共享决策的情况下生成兼容代码的难度。这些结果支持了多智能体代码生成的“规范优先”观点：更丰富的规范说明既是主要的协作机制，也是充分的恢复手段。

## Abstract
When multiple LLM-based code agents independently implement parts of the same class, they must agree on shared internal representations, even when the specification leaves those choices implicit. We study this coordination problem across 51 class-generation tasks, progressively stripping specification detail from full docstrings (L0) to bare signatures (L3), and introducing opposing structural biases (lists vs. dictionaries) to stress-test integration. Three findings emerge. First, a persistent specification gap: two-agent integration accuracy drops from 58% to 25% as detail is removed, while a single-agent baseline degrades more gracefully (89% to 56%), leaving a 25--39 pp coordination gap that is consistent across two Claude models (Sonnet, Haiku) and three independent runs. Second, an AST-based conflict detector achieves 97% precision at the weakest specification level without additional LLM calls, yet a factorial recovery experiment shows that restoring the full specification alone recovers the single-agent ceiling (89%), while providing conflict reports adds no measurable benefit. Third, decomposing the gap into coordination cost (+16 pp) and information asymmetry (+11 pp) suggests that the two effects are independent and approximately additive. The gap is not merely a consequence of hidden information, but reflects the difficulty of producing compatible code without shared decisions. These results support a specification-first view of multi-agent code generation: richer specifications are both the primary coordination mechanism and the sufficient recovery instrument.

### 论文总结：规范差距——代码智能体在部分知识下的协作失败

#### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）在代码生成领域的应用从单函数转向复杂的多智能体协作系统，一个新的挑战浮出水面：**协调问题（Coordination Problem）**。当多个智能体独立实现同一个类的不同方法时，它们必须在共享的内部状态（如构造函数中初始化的数据结构）上达成一致。如果规范说明（Specification）不够明确，不同智能体可能会基于各自的偏好做出不兼容的选择（例如一个用列表，另一个用字典），导致集成失败。本文旨在量化这种由于“部分知识”导致的“规范差距”，并探讨如何有效修复这种协作失败。

#### 2. 方法论：核心思想与关键技术
研究者构建了一个受控实验框架，通过操纵规范的完整性来观察其对多智能体协作的影响：
*   **规范等级划分（L0-L3）：**
    *   **L0（完整）：** 包含详细文档字符串、Doctest 示例和明确的数据结构引用。
    *   **L1（简化）：** 移除 Doctest，保留数据结构引用。
    *   **L2（抽象）：** 移除数据结构引用，使用抽象语言描述行为。
    *   **L3（极简）：** 仅保留函数签名。
*   **偏差注入（Bias Injection）：** 为了模拟现实中开发者的不同风格，研究者给智能体 A 注入“偏好列表”的偏差，给智能体 B 注入“偏好字典”的偏差。
*   **AST 冲突检测器：** 开发了一个基于抽象语法树（AST）的检测器，无需 LLM 调用即可识别类型冲突（如 `append` 与 `key-value` 赋值冲突）和状态访问冲突。
*   **恢复机制：** 引入“合并智能体（Merger Agent）”，测试在提供不同信息（完整规范 vs. 冲突报告）时，其修复冲突代码的能力。

#### 3. 实验设计
*   **数据集：** 基于 **ClassEval** 基准测试，筛选出 51 个具有 3 个以上方法的类生成任务，构建了名为 **AmbigClass** 的变体数据集。
*   **对比场景：**
    *   **Single(ℓ)：** 单智能体基准（性能上限），可见构造函数。
    *   **Split(ℓ)：** 双智能体协作，不可见构造函数，且带有相反的结构偏差。
*   **模型：** 主要使用 Claude 3.5 Sonnet，并使用 Claude 3.5 Haiku 进行跨模型验证。
*   **评估指标：** 单元测试通过率（Pass Rate）和 AST 检测到的冲突数量。

*   **算力说明：** 论文未提及具体的 GPU 训练算力，因为该研究基于现有的商业模型 API 进行推理实验。

#### 5. 实验数量与充分性
*   **实验规模：** 涵盖 51 个任务、4 个规范等级、2 种模型、多次独立运行。
*   **消融与析因分析：**
    *   **2x2 恢复实验：** 交叉验证“规范质量”与“冲突报告”对修复效果的贡献。
    *   **2x2 初始化可见性实验：** 将性能差距分解为“协作成本”和“信息不对称”两部分。

#### 6. 主要结论与发现
*   **规范差距显著：** 随着规范细节减少，多智能体集成成功率从 58% 骤降至 25%，与单智能体相比存在 **25-39 个百分点** 的持久差距。
*   **协调税（Coordination Tax）：** 即使在 L0 完整规范下，多智能体表现仍远低于单智能体，说明独立决策本身就带有难以消除的协调成本。
*   **规范是核心恢复工具：** 实验证明，只要给合并智能体提供完整的 L0 规范，就能恢复到单智能体的性能水平（89%）；而单纯提供冲突报告而不提供规范，对修复几乎没有帮助。
*   **差距分解：** 性能差距由协作成本（约 16pp）和信息不对称（约 11pp）共同构成，两者具有近似可加性。

#### 7. 优点（亮点）
*   **理论与实践结合：** 将经典的软件工程理论（如 Parnas 的信息隐藏、Meyer 的契约式设计）应用于现代 AI 智能体研究。
*   **零成本诊断：** 提出的 AST 检测器在弱规范下精确度高达 97%，为监控多智能体协作质量提供了一种高效、廉价的手段。
*   **严谨的实验分解：** 通过巧妙的 2x2 实验设计，成功剥离了“看不见代码”和“难以协作”这两个容易混淆的因素。

#### 8. 不足与局限
*   **语言与任务单一：** 实验仅限于 Python 语言和类级别的生成任务，尚未验证在其他编程范式或更大规模系统（如跨文件协作）中的表现。
